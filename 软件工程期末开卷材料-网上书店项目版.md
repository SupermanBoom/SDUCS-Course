## 1. 敏捷开发相关大题

### 1.1 敏捷开发的定义

敏捷开发以**用户需求持续变化**为核心，采用**迭代、增量**的方式开发软件，将项目拆分为多个短周期迭代，每次迭代都交付一个**可运行、可测试、可集成**的软件版本。

### 1.2 敏捷开发相对于传统开发的优点

1. 能快速响应需求变化
2. 用户能尽早看到可运行版本
3. 团队沟通频繁，反馈及时
4. 风险暴露较早，便于及时调整
5. 适合需求不完全明确或变化较快的项目

### 1.3 敏捷开发相对于传统开发的缺点

1. 对团队协作、自律和沟通能力要求较高
2. 文档可能不如瀑布模型完整
3. 项目早期难以精确估算成本和工期
4. 大型团队或强监管项目中管理难度较高

### 1.4 敏捷开发三大核心理念

1. 聚焦客户价值，减少不必要的浪费
2. 激发团队潜能，强调高效协作
3. 持续迭代调整，主动适应需求变化

### 1.5 三种敏捷实践方法

#### 1. Scrum

以 Sprint 为周期进行迭代开发。角色包括：

1. 产品负责人 Product Owner
2. Scrum Master
3. 开发团队

常见活动包括：

1. 产品待办列表
2. 冲刺计划会
3. 每日站会
4. 评审会
5. 回顾会

#### 2. XP 极限编程

强调：

1. 结对编程
2. 测试驱动开发
3. 持续集成
4. 重构
5. 小版本发布

#### 3. Kanban 看板

通过“待办 / 进行中 / 已完成”等列可视化任务状态，限制在制品数量，提高流程透明度和交付效率。

### 1.6 结合网上书店项目作答模板

本项目更适合采用 **Scrum**。因为网上书店系统可以自然拆分为多个迭代，如“用户注册登录”“图书管理”“购物车”“订单支付”“后台管理”等模块。每个 Sprint 完成一个可运行的子系统，便于教师或用户及时提出修改意见，也有利于团队边开发边测试边改进。

### 1.7 若题目问“简述敏捷开发优缺点，并说明项目为什么适合敏捷”

可直接写：

敏捷开发强调小步迭代、快速交付和持续反馈。与传统瀑布开发相比，敏捷开发能更快响应需求变化，让用户尽早看到可运行软件，适合需求不稳定的项目；但它对团队协作和自律要求较高，文档相对较弱。网上书店项目功能较多，且前台展示、购物车、订单等功能常需要根据使用体验调整，因此适合采用敏捷方式分阶段迭代开发。

1. 名称

## 3. 项目计划、进度管理与估算

### 3.1 跟踪项目进度的两种工具

#### 1. 活动图

作用：

1. 表达活动之间的先后依赖关系
2. 用于分析并行关系
3. 计算关键路径
4. 预估总工期

#### 2. 甘特图

作用：

1. 直观展示每项任务的开始时间和结束时间
2. 显示并行任务
3. 方便日常项目跟踪
4. 可标识延期和关键任务

### 3.2 关键路径计算题模板

#### 标准四步

1. 列出所有顶点，计算每个事件的最早发生时间 `ve`
2. 从终点逆推，计算每个事件的最迟发生时间 `vl`
3. 对每个活动 `a(i,j)` 计算：
   - 最早开始时间 `e = ve(i)`
   - 最迟开始时间 `l = vl(j) - dur(i,j)`
4. 找出满足 `e = l` 的活动，这些活动组成关键路径

#### 最终答案写法

1. 关键活动为：……
2. 关键路径为：……
3. 总工期为：……

### 3.3 Delphi 专家估算 / 三点估算

若考简答，可写：

Delphi 估算法是通过多位专家匿名多轮给出估计值，并逐步趋于一致的一种经验估算方法。其优点是能综合多位专家经验，减少个人偏差，适用于需求尚不完全明确的项目估算。

### 3.4 决策树 EMV 期望收益

公式：

`单项期望 = 发生概率 × 对应收益或损失`

`方案总 EMV = 所有分支期望之和`

比较多个方案时，通常选择 **EMV 最大** 的方案。

### 3.5 WBS 工作分解结构

定义：
WBS 是将项目整体范围逐层细化拆分，直到形成可执行、可管理的最小工作单元。

三种分解方法：

1. 类比法：参考类似项目模板
2. 自顶向下法：从总体任务逐层细化
3. 自底向上法：先列出具体工作，再向上归纳整合

### 3.6 结合网上书店项目的 WBS 示例

可分解为：

1. 需求分析
2. 数据库设计
3. 用户模块开发
4. 图书模块开发
5. 购物车模块开发
6. 订单与支付模块开发
7. 后台管理模块开发
8. 测试与部署

### 3.7 软件规模和工作量估算

#### 两种主流规模度量

1. KLOC：代码行数，适合底层或系统类软件
2. FP：功能点，适合业务系统和管理软件

#### 常用工作量估算方法

1. 专家判断法
2. 类比估算法
3. 静态单变量模型
4. COCOMO 成本模型
5. 回归模型 / 神经网络等量化方法

### 3.8 若题目问“结合项目如何跟踪进展”

可写：

在网上书店项目中，可先利用 WBS 将项目拆分为用户、图书、购物车、订单、支付、后台等任务，再利用甘特图安排每个任务的起止时间，用活动图分析模块依赖和关键路径，从而实现对项目进度的持续跟踪和控制。

## 4. 需求工程大题

### 4.1 完整需求开发四步过程

1. 需求获取 Elicitation：收集用户原始需求
2. 需求分析 Analysis：理解、整理、建模需求
3. 需求规格说明 Specification：形成正式需求文档
4. 需求验证 Validation：检查需求是否真实、完整、正确

最终产出物：**SRS 软件需求规格说明书**

### 4.2 三类需求及网上书店项目例子

#### 1. 功能需求

定义：系统应当提供的服务、输入输出行为以及在特定条件下系统应采取的动作。

网上书店项目例子：

1. 用户能够注册和登录系统
2. 用户能够按书名、作者、分类搜索图书
3. 用户能够将图书加入购物车并修改购买数量
4. 用户能够提交订单并完成支付
5. 管理员能够新增、修改、删除图书信息

#### 2. 非功能需求

定义：系统整体的限制性要求和全局质量属性，如性能、可靠性、安全性、可维护性、易用性等。

网上书店项目例子：

1. 系统能支持多个用户同时访问
2. 支付过程应保证数据安全
3. 页面响应时间应控制在可接受范围内
4. 系统应具备一定的容错和恢复能力

#### 3. 领域需求

定义：来源于特定业务领域的规则和约束。

网上书店项目例子：

1. 订单只有在支付成功后才能进入待发货状态
2. 库存不足时不能完成下单
3. 订单取消后应恢复对应库存

### 4.3 若题目要求“结合项目阐述两个功能性需求及建模过程”

可直接写：

功能需求一：用户输入书名、作者或分类后，系统应返回符合条件的图书列表。
建模过程：先在需求获取阶段通过访谈明确查询方式，在分析阶段抽象为“搜索图书”用例，再在规格说明阶段写明输入、处理和输出，最后通过原型或评审进行验证。

功能需求二：用户提交订单并支付成功后，系统应生成订单记录、修改订单状态并扣减库存。
建模过程：在分析阶段识别参与者“用户”和“系统”，抽象出“提交订单”“支付订单”“更新库存”等用例，并使用状态图描述订单从“未支付”到“已支付”的变化过程。

### 4.4 常见建模图

1. E-R 图：描述实体及联系
2. UML 类图：描述类、属性、方法与关系
3. DFD 数据流图：描述数据如何流动
4. UML 状态图：描述对象状态变化

### 4.5 抛弃型原型 vs 演化型原型

#### 1. 抛弃型原型

核心：仅用于澄清需求，需求确认后原型丢弃，不作为最终系统一部分。

优点：

1. 开发快
2. 成本低
3. 适合验证界面和模糊需求

缺点：

1. 代码不能复用
2. 最终系统需要重新开发

适用场景：

1. 需求很模糊
2. 客户不清楚界面或流程

#### 2. 演化型原型

核心：原型在不断改进后直接演变为最终系统。

优点：

1. 代码可复用
2. 可持续交付
3. 用户可持续验收

缺点：

1. 初期就要考虑架构和性能
2. 早期设计缺陷可能长期保留

适用场景：

1. 需求逐步清晰
2. 需要长期持续迭代的软件

### 4.6 用例之间 include 与 extend 的区别

#### 1. `<<include>>`

含义：基础用例**必须**执行被包含用例。
作用：抽取公共步骤。
例子：在网上书店中，“提交订单”可以 `include` “用户身份验证”。

#### 2. `<<extend>>`

含义：扩展用例是**可选的**，只有满足条件时才执行。
作用：在基础功能上添加可选附加行为。
例子：在网上书店中，“使用优惠券”可以 `extend` “提交订单”。

## 5. 软件体系结构与系统分解

### 5.1 软件体系结构设计过程

1. 架构建模
2. 初步架构分析
3. 架构评估实验
4. 记录架构决策
5. 评审校验架构是否满足需求
6. 输出软件体系结构

### 5.2 六种主流系统分解方法

1. 面向功能分解：按功能拆模块
2. 面向特征分解：按业务特征拆模块
3. 面向数据分解：围绕数据组织模块
4. 面向进程分解：按并发任务或进程拆分
5. 面向事件分解：按输入事件及状态转换拆分
6. 面向对象分解：按对象及对象关系拆分

### 5.3 三种常见软件体系结构风格

#### 1. 分层体系结构

系统分为表示层、业务层、数据访问层、数据层。
优点：结构清晰、易维护、易扩展。
缺点：层间调用可能增加性能开销。

#### 2. C/S 结构

客户端负责交互，服务器负责业务处理和数据管理。
优点：交互能力强。
缺点：客户端安装和升级成本较高。

#### 3. MVC 结构

1. Model：数据和业务对象
2. View：页面显示
3. Controller：请求分发和业务流程控制

优点：表现层、控制层、数据层分离，利于维护和复用。

### 5.4 结合网上书店项目说明采用哪种结构

本项目采用 **Java Web 的 MVC + 分层结构**。

1. View 层：JSP 页面，如首页、购物车页、后台图书列表页
2. Controller 层：Servlet，如 `UserServlet`、`CartServlet`、`OrderServlet`
3. Model / DAO 层：`Book`、`User`、`Order` 等实体类以及 `BookDao`、`OrderDao`
4. Data 层：MySQL 数据库，表如用户表、图书表、订单表、订单项表

### 5.5 若题目问“为什么项目适合 MVC + 分层”

可写：

网上书店既有前台页面展示，又有较复杂的业务处理和数据库访问，因此采用 MVC + 分层结构最合适。这样可以将页面显示、流程控制、业务逻辑和数据存储分离，既便于多人协作开发，也有利于后续维护和功能扩展。

## 6. 模块设计、文档化与编码规范

### 6.1 源程序文档化的作用

程序的阅读和维护时间远长于编写时间，因此程序文档化能够显著降低维护成本，提高代码可理解性。

源程序文档化主要包括三部分：

1. 标识符命名
2. 注释
3. 视觉组织

### 6.2 标识符命名规范

1. 见名知意，名称能体现实体含义
2. 长度适中，可合理缩写
3. 不使用容易混淆的相似名称
4. 不使用关键字
5. 同一标识符只能表示一种含义
6. 避免易混淆字符，如 `0/O`、`1/I/l`

### 6.3 程序注释的两类

#### 1. 序言性注释

位于模块头部，说明：

1. 模块功能和用途
2. 接口信息
3. 输入输出
4. 依赖关系
5. 作者、日期、修改记录

#### 2. 功能性注释

位于程序中间，说明代码段意图和逻辑。

要求：

1. 注释准确
2. 程序修改时注释同步更新
3. 注释描述“为什么做”，不重复代码字面含义
4. 注释以逻辑块为单位，不必逐行注释

### 6.4 视觉组织

1. 缩进：表现层次结构
2. 空行：分隔逻辑块
3. 空格：增强可读性
4. 括号：明确优先级
5. 大括号风格统一

### 6.5 数据说明规范

1. 固定说明顺序：常量 → 自定义类型 → 全局变量 → 局部变量
2. 一条语句中多个变量按字典序排列
3. 复杂数据结构必须有说明性注释

### 6.6 语句构造原则

1. 清晰优先于技巧
2. 一行只写一条语句
3. 减少多层嵌套
4. 复杂表达式用括号
5. 优先使用标准库函数
6. 少用 `goto`
7. 变量使用前先初始化

### 6.7 输入输出设计原则

1. 输入格式和操作步骤尽量简化
2. 做好输入合法性校验
3. 报错信息明确
4. 统一输入格式
5. 输出结果应整洁、易读、可视化

### 6.8 程序效率准则

1. 效率指标属于性能需求，应在需求阶段提出
2. 正确、清晰优先于微小效率提升
3. 简单优化交给编译器
4. 根本优化来自好的架构、数据结构和算法

### 6.9 结合网上书店项目举例

可写：

在网上书店项目中，应将用户、图书、订单、库存等模块分别命名清晰，例如 `UserService`、`BookDao`、`OrderItem` 等；对订单提交流程、库存扣减逻辑、支付状态更新等关键代码段添加功能性注释；在购物车金额计算和订单状态流转等复杂逻辑中，采用统一缩进、分块和括号来提高可读性。

## 7. 面向对象设计原则（结合网上书店项目）

### 7.1 单一职责原则

定义：一个类或模块只应有一个引起它变化的原因，即只负责一项职责。

项目应用：

1. `UserService` 负责用户注册、登录、信息维护
2. `BookService` 负责图书查询和管理
3. `OrderService` 负责订单创建、取消、查询
4. `InventoryService` 负责库存扣减和恢复

作用：职责清晰，修改某一类功能时不容易影响其他模块。

### 7.2 开闭原则

定义：软件实体应对扩展开放、对修改关闭。

项目应用：

在支付模块中可定义统一支付接口，如 `Payment`，支付宝、微信、银行卡分别实现该接口。以后新增新的支付方式时，只需增加新实现类，不需要改动原有业务流程。

作用：提高系统扩展性。

### 7.3 里氏代换原则

定义：子类对象必须能够替换父类对象，并且不影响程序的正确性。

项目应用：

定义 `Book` 父类，再设计 `PaperBook` 和 `EBook` 子类。在图书展示、加入购物车、价格计算等处理中，可以统一把它们当作 `Book` 使用。

作用：提高复用性和通用性。

### 7.4 依赖倒置原则

定义：高层模块不依赖低层模块，二者都依赖抽象；抽象不依赖细节，细节依赖抽象。

项目应用：

订单模块不应直接依赖具体支付实现类，而应依赖支付接口。
例如：`OrderService` 依赖 `Payment` 接口，而不是直接依赖 `AliPayService`。

作用：降低耦合，便于替换实现。

### 7.5 接口隔离原则

定义：客户端不应依赖它不需要的接口，应当把大接口拆分为多个小接口。

项目应用：

普通用户接口与管理员接口分开设计：

1. 普通用户：浏览图书、加入购物车、下单、支付
2. 管理员：图书上架、下架、修改库存、查看报表

作用：避免接口臃肿，提高设计合理性。

### 7.6 合成复用原则

定义：尽量使用组合或聚合复用，而不是单纯依赖继承。

项目应用：

1. 购物车由多个 `CartItem` 组合而成
2. 订单由 `OrderItem`、`Address`、`PaymentInfo` 等组合而成

作用：结构更灵活，扩展性更好。

### 7.7 迪米特原则

定义：一个对象应尽量少了解其他对象，只与直接朋友通信，也称最少知识原则。

项目应用：

前端页面提交订单时，不直接操作库存模块和支付模块，而是统一调用 `OrderService`，由 `OrderService` 协调库存、支付和订单持久化流程。

作用：减少调用链复杂度，降低模块间耦合。

### 7.8 原则大题收尾模板

在网上书店项目中，面向对象设计原则主要体现在模块划分清晰、接口设计合理、业务扩展方便等方面。通过遵循单一职责、开闭、依赖倒置等原则，可以提高系统的可维护性、可扩展性和可复用性。

## 8. 面向对象设计模式（结合网上书店项目）

### 8.1 单例模式

定义：保证一个类只有一个实例，并提供全局访问点。

项目应用：

1. 系统配置管理类
2. 数据库连接池管理类
3. 日志管理类

作用：节省资源，保证全局一致性。

### 8.2 建造者模式

定义：将一个复杂对象的构建过程与其表示分离，使同样的构建过程可以创建不同表示。

项目应用：

订单对象较复杂，包含：

1. 用户信息
2. 订单项列表
3. 收货地址
4. 总金额
5. 优惠信息
6. 支付方式

可使用建造者模式分步骤构造订单对象。

作用：使复杂对象创建过程清晰。

### 8.3 观察者模式

定义：定义对象间一对多依赖关系，当一个对象状态变化时，所有依赖它的对象都会收到通知并自动更新。

项目应用：

支付成功后，订单状态发生变化，此时可通知：

1. 库存模块扣减库存
2. 积分模块增加积分
3. 消息模块发送通知

作用：降低模块之间的硬编码依赖，提高扩展性。

### 8.4 中介者模式

定义：用一个中介对象封装多个对象之间的交互，使各对象不直接相互引用。

项目应用：

订单处理涉及购物车、库存、支付、优惠券、物流等多个模块，可由 `OrderMediator` 统一协调各模块交互。

作用：减少对象间网状调用，降低耦合。

### 8.5 MVC 模式

定义：Model 负责数据和业务对象，View 负责显示，Controller 负责控制流程。

项目应用：

1. JSP 是 View
2. Servlet 是 Controller
3. Bean / DAO / Service 是 Model

### 8.6 模板方法模式

定义：在父类中定义算法框架，在子类中实现具体步骤。

项目应用：

Servlet 继承 `HttpServlet`，由 Web 容器调用 `doGet()`、`doPost()`，具体业务由各个 Servlet 重写实现。

### 8.7 设计模式题收尾模板

本项目在整体上采用了 MVC 架构模式，在局部设计中可体现单例、建造者、观察者和中介者等设计模式。这些模式分别解决了对象唯一管理、复杂对象创建、状态联动通知和多模块协同的问题。

## 9. 耦合度与内聚度分析

### 9.1 耦合度

定义：模块之间相互依赖的紧密程度。耦合度越低越好。

项目分析：

如果订单模块直接操作库存、支付、用户、通知等多个模块的内部细节，那么模块之间联系过多，属于高耦合。更合理的做法是通过 `OrderService`、`PaymentService`、`InventoryService` 等接口或服务层完成协作。

结论：网上书店系统应追求**低耦合**。

### 9.2 内聚度

定义：一个模块内部各组成部分之间联系的紧密程度。内聚度越高越好。

项目分析：

如果 `OrderService` 只负责创建订单、取消订单、查询订单和修改订单状态，则说明其功能集中、内聚度高；如果它同时处理用户登录、图书管理、库存修改等无关功能，则内聚度低。

结论：网上书店系统应追求**高内聚**。

### 9.3 综合结论

网上书店项目在设计时应遵循“**高内聚、低耦合**”原则，即每个模块内部功能尽量集中，模块之间尽量通过接口和服务进行协作，而不直接依赖对方内部实现。

## 10. 白盒测试与黑盒测试

### 10.1 白盒测试五种覆盖标准

按覆盖强度由弱到强：

1. 语句覆盖
2. 判定覆盖（分支覆盖）
3. 条件覆盖
4. 判定-条件覆盖
5. 条件组合覆盖

### 10.2 各覆盖标准定义

1. 语句覆盖：所有语句至少执行一次
2. 判定覆盖：每个判定的真分支和假分支都至少执行一次
3. 条件覆盖：每个独立条件都至少取真、假一次
4. 判定-条件覆盖：同时满足判定覆盖和条件覆盖
5. 条件组合覆盖：一个判定中所有条件真假组合都覆盖

### 10.3 白盒计算题通用步骤

1. 找出所有判定
2. 拆出每个独立条件
3. 根据覆盖标准设计测试输入
4. 分析路径和输出结果

### 10.4 黑盒测试四大方法

#### 1. 等价类划分法

将输入域划分为有效等价类和无效等价类，同类中取一个代表值即可。

网上书店项目例子：

注册密码长度要求 6~16 位。

1. 有效类：6~16 位合法字符
2. 无效类：长度小于 6、长度大于 16、含非法字符

#### 2. 边界值分析法

重点测试边界值及其附近值。

网上书店项目例子：

购买数量范围为 1~99，可测试：

1. 0
2. 1
3. 2
4. 98
5. 99
6. 100

#### 3. 错误推测法

基于经验推测程序容易出错的位置。

网上书店项目例子：

1. 库存为 0 时仍能下单
2. 支付成功后订单状态没有更新
3. 重复点击支付按钮导致重复扣款

#### 4. 因果图法

适合分析多个输入条件与输出结果的逻辑关系。

网上书店项目例子：

分析“会员折扣 + 优惠券 + 满减活动”共同作用下的订单结算结果。

## 11. 集成测试、系统测试与项目测试大题

### 11.1 单元测试、集成测试、系统测试对比

#### 1. 单元测试

测试对象：单个模块
目标：发现模块内部逻辑错误
依据：详细设计说明书

#### 2. 集成测试

测试对象：模块之间的接口与协作
目标：发现接口错误、数据传递错误、全局数据问题
依据：软件体系结构设计

#### 3. 系统测试

测试对象：完整软件系统
目标：验证整体功能、性能、安全性等是否满足需求
依据：软件需求规格说明书

### 11.2 两个基础概念

1. Driver 驱动模块：模拟上层调用者
2. Stub 桩模块：模拟下层被调用模块

### 11.3 集成测试策略

#### 1. 大爆炸集成

全部模块完成后一次性集成测试。
优点：简单。
缺点：故障定位困难，只适合小系统。

#### 2. 自顶向下集成

从顶层模块开始，逐步用真实模块替换桩模块。
优点：可以较早验证整体控制流程。
缺点：需要较多桩模块。

#### 3. 自底向上集成

从底层叶子模块开始，逐步向上集成，借助驱动模块测试。
优点：底层模块可较早验证，故障定位清晰。
缺点：高层流程验证较晚。

#### 4. 三明治集成

上层自顶向下、下层自底向上同步进行。
优点：兼顾两者优点。
缺点：中间层协调复杂。

### 11.4 结合网上书店项目如何回答“如何对软件进行测试（过程、方法、文档）”

## 12. 故障分析、可信度与可用性

### 12.1 错误播种法

设：

1. `Ns`：人为植入故障数
2. `ns`：测试中发现的植入故障数
3. `n`：测试中发现的原有故障数

则原有故障总数估计为：

`N = Ns × n / ns`

剩余故障数估计为：

`Nr = N - n`

### 12.2 可信度公式

`R(t) = e^(-λt)`

其中：

1. `λ` 为单位时间故障率
2. `t` 为运行时间

### 12.3 可用性公式

`A = MTTF / (MTTF + MTTR)`

其中：

1. `MTTF`：平均无故障时间
2. `MTTR`：平均修复时间

### 12.4 答题注意

若是计算题，最后一定要写出：

1. 计算公式
2. 代入过程
3. 结果
4. 单位或百分比
5. 一句解释性结论

## 13. 最小割集与故障树分析

### 13.1 最小割集定义

在故障树中，能够导致顶事件发生的一组基本事件称为割集。
其中，去掉任一基本事件后就不再构成割集的，称为最小割集。

### 13.2 一般求法

1. 自顶向下展开故障树
2. 遇与门取“乘”
3. 遇或门取“并”
4. 列出所有割集
5. 化简并删除被包含的大集合，保留最小集合

### 13.3 最终答案写法

最小割集为：`{X1, X2}`、`{X3}`、`{X4, X5}`
